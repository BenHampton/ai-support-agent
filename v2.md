# V2 — Deferred Scope & Tradeoffs

Work intentionally **not** done in v1, with the reasoning. This is the durable home for "we decided
not to do X yet, and here's why" — a roadmap + tradeoffs record, not a bug list.

**Guiding principle used throughout:** ship the *smallest correct fix* now, under a regression net,
and defer the robust/expensive version with an explicit rationale. Never tune a fuzzy system (rules,
thresholds, prompts) blind — build the eval harness first, then change behavior against it. Deferral
here is scoping, not neglect: every item below was a conscious call, not an oversight.

## 1. Grounding-aware routing  *(flagship)*

**The class of bug.** Routing is keyword-based (`rules.yaml` keyword sets + `engine.ts`). A rule can
fire because a keyword *appears*, even when the knowledge base has no real answer. Found via
red-teaming: `consumer-us` asking *"What's your 90-day price-match guarantee?"* was routed by
`selfServeBillingRule` and labeled *"Non-VIP billing question — self-serve/CSM"*, purely because
"price" was in the `billing` keyword set. The customer reply was fine (prompt boundaries stopped the
hallucination), but the **decision label was wrong** — an observability defect a support lead would
act on.

**What we shipped now (the small fix).** Removed the ambiguous `price` / `cost` / `fee` from the
`billing` keyword set, and encoded the failure as a permanent regression case in the eval harness
(`packages/api/src/rules/engine.test.ts`, `scenarios.eval.test.ts`).
- *Why now:* cheap, low-risk, fixes the observed instance, **verifiable under the new harness**
  (it failed red → passed green, and the 6 demo scenarios stayed green), and it unblocks the demo.

**Why we did NOT do the robust fix now.** The robust fix is an **LLM grounding check**: after
retrieval, ask a small model *"does the retrieved KB actually answer this question? yes/no"* — and if
no, escalate (or answer an honest "we don't offer that"), regardless of which keyword matched. That
is the only thing that reliably separates *"the sentence contains the word price"* from *"this is an
answerable billing question."* But it:
- requires **Ollama + an extra model call per request** (latency + cost),
- needs **tuning** against real traffic to set the escalate threshold,
- needs its own **answer-quality evals** to validate (which we also don't have yet).

It's a meaningful build, not a demo-blocker — and doing it *before* the harness existed would have
been tuning blind.

**Why it moves to v2.** The keyword fix plus this documented tradeoff already demonstrates the
engineering judgment. The robust fix is higher-cost and genuinely wants real usage data to tune. With
the harness now in place, it can be built and measured safely later. This is scoping, not neglect.

**Residual risk (stated honestly).** Keyword routing is whack-a-mole. Other ambiguous words —
"quote," "estimate," "plan" — can re-trigger the same class of misclassification. The harness catches
regressions on **known** cases; it does not catch **new** blind spots. The small fix fixed the
*instance*; the grounding check fixes the *class*.

**What v2 looks like.** A grounding/relevance check between retrieval and answer (heuristic threshold
first, LLM-judge for quality), escalating on ungrounded questions. It doubles as hardening **Step 3
(output validation)** — same mechanism, so build once, use for both.

## 2. Output validation  *(hardening Step 3)*

Today, model tokens stream straight to the user with **no post-generation checks** (`ollama.ts` →
`orchestration.ts`). Deferred because it overlaps the grounding fix above and needs its own eval
cases to be meaningful. v2 adds: a leak check (reply doesn't echo the system prompt), and a grounding
check (answer is supported by the retrieved KB). See `prompt-hardening.md` Step 3.

## 3. Answer-quality evals (LLM-as-judge)

The eval harness we built covers **decision correctness** (right decision + right rule),
deterministically, with no Ollama. It does **not** grade **answer quality** (is the reply accurate,
grounded, on-tone?). That needs the LLM plus an LLM-judge and is nondeterministic. Deferred as the
second dimension of the harness.

## 4. End-to-end evals (embed → retrieve → rules)

Current scenario evals run at the **engine level** with hand-authored `knowledgeMatches`, so they're
deterministic and test the rules engine — not retrieval. A full-pipeline eval (real embeddings +
cosine search + rules) would also exercise retrieval quality, but needs Ollama and is slower/flakier.
Deferred as an Ollama-gated integration suite.

## 5. Failure-mode / resilience

Only a single outer `try/catch` exists (`routes/chat.ts`); any dependency failure — Salesforce,
Zendesk, or Ollama — collapses into one generic `{ type: 'error' }` SSE frame. No graceful
degradation. (Spec line 41: "Failure modes, retries, fallbacks, and rollback paths.") v2 adds
per-dependency fallbacks: Salesforce down → KB-only answer or safe escalate; Zendesk down → don't
drop the escalation (queue/retry, surface an alternate channel); Ollama down → fall back to escalate;
plus typed errors and trace records of the degraded path. Note: this is *integration* outage
handling — distinct from the *product* outage feature (ArkCloud status), which is built.

## 6. Rule fidelity vs spec

- **`refundEligibilityRule`** checks only `region` + `purchaseDate`. The spec calls for **account
  status + product type** too (2 of 4 factors present). Deferred — needs the product policy defined.
- **`regulatedTopicRule`** sets `requiresComplianceLanguage: true`, but that flag is **not consumed
  anywhere** in `orchestration.ts` (dead signal), and there's **no human-review** step. Spec asks for
  "approved language *and* human review." v2: consume the flag in the prompt and add a review handoff.

## 7. Remaining hardening steps (5–7)

From `prompt-hardening.md`, not yet done (steps 1, 2, 4 are done):
- **Step 5 — sanitize downstream.** The raw user message flows into the Zendesk ticket's
  `conversationContext` unescaped — a downstream-injection sink once Zendesk is real.
- **Step 6 — least privilege for context.** The full customer record is injected into the prompt;
  trim to what's needed (tier/region/products).
- **Step 7 — guardrails.** No input length cap, no rate limiting.

## 8. Smaller items

- **UI loading/error states** — `SessionList` fetches with a bare `.then()`, no catch; a failed
  fetch silently shows an empty table (violates the UI `CLAUDE.md` "explicit loading & error states"
  convention).
- **Outage toggle live-reload** — `OUTAGE_ACTIVE` is read at process start, so toggling requires a
  restart. Fine for a demo; a real status API would be live.
- **Doc drift** — `packages/api/CLAUDE.md` says lowConfidence threshold 0.4 (yaml is 0.5) and top-3
  KB chunks (README/code say top-5). Batch-fix later.
- **Eval harness in CI** — the suite runs locally (`npm test`); wire it into a GitHub Action so
  routing/decision regressions are caught on every PR.
